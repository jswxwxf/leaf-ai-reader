# PRD 004: 网页文章采集与 D1 结构化存储

## 1. 概述
为 Leaf AI Reader 引入“文章 (Article)”分类。该功能集成在 D1 数据库系统的 `articles` 表中，用于存储用户日常采集的网络文章。
**核心优势**：利用 D1 的关系型存储提供强一致性状态管理，支持大规模文章存储及未来的高级分页/搜索功能。

## 2. 核心功能设计

### 2.1 采集逻辑：双模驱动 (Dual-Mode)

#### A. 自动模式 (Auto Fetch)
*   **触发**：用户输入网页 URL。
*   **后端流**：`fetch` (带仿冒 Headers) -> `linkedom` (DOM 解析) -> `@mozilla/readability` (正文提取)。
*   **失败处理**：若返回 403、验证码或内容为空，系统自动切换至“手动模式”。

#### B. 手动模式 (Manual Fallback)
*   **场景**：自动抓取由于反爬、登录墙而失败。
*   **操作**：前端提供粘贴区域，引导用户在原网页“全选并复制”，然后“在此处粘贴”。
*   **处理**：提取粘贴内容中的纯净正文，流程同自动模式。

### 2.2 存储方案 (混合存储)
为应对大规模数据及 D1 的存储限制，采取混合存储策略：

*   **D1 数据库 (`articles` 表)**: 存储元数据、状态管理及短小内容。
    - `id`: TEXT PRIMARY KEY (UUID)
    - `user_id`: TEXT (用户唯一标识，带索引)
    - `title`: TEXT (文章标题)
    - `status`: TEXT (processing | ready | error)
    - `source_url`: TEXT (原始 URL)
    - `content`: TEXT (**过渡期方案**：暂存清理后的 HTML，受 D1 单行 2MB 限制)
    - `created_at`: DATETIME (默认当前时间)
*   **R2 存储桶 (正文持久化 - 计划中)**:
    - **路径**: `articles/${userId}/${articleId}/content.html`
    - **原因**: D1 单行限制 2MB，且 SQL 语句长度限 100KB。对于超长 HTML 页面，必须迁移至 R2 以确保稳定性。

### 2.3 列表管理机制
*   **逻辑**：
    1. 前端发起请求，API 立即在 D1 中创建一条 `processing` 状态的记录。
    2. 使用 `ctx.waitUntil` 异步触发后台 Worker 进行实际的抓取和解析。
    3. 后端查询时按时间倒序排列，实现 LIFO 效果。
*   **分阶段实现**：
    - **当前阶段**：展示所有采集文章。
    - **未来阶段**：实现基于数据库的分页加载，以应对海量数据。

### 2.4 图片处理
*   **Proxy 模式**：通过 `api/proxy-img?url=...` 动态转发图片，剥离 `Referer` 以突破防盗链。
*   **存储优化**：数据库仅存储正文 HTML，图片依然通过原始 URL 动态加载或 Proxy 转发。

## 3. UI/UX 表现
*   **入口**：Dashboard 增加“采集文章”按钮。
*   **展示**：在 Dashboard 的文章视图中展示已采集列表。
*   **状态**：显示“正在采集...”动效，Worker 异步解析成功后前端通过轮询感知状态并允许进入阅读。
