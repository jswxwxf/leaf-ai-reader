# PRD 014: 摘要手动同步重刷新机制

## 1. 背景与目的
在 AI 摘要生成失败、内容缺失或用户对结果不满意时，需要提供一种简单的机制来强制重新触发 AI 生成流程。为了保持代码简洁并提供直观反馈，采用“同步触发 + 覆盖更新”的方案。

## 2. 核心交互流程
1.  **触发阶段**：用户点击摘要栏中的“重新生成”按钮。
2.  **执行阶段**：
    *   按钮进入加载状态（Loading）。
    *   前端向后端发送 `POST /api/reader/summarize` 请求。
    *   后端接口同步调用 Worker 的处理方法（`processChapter` 或 `processArticle`）并使用 `await` 等待。
3.  **覆盖阶段**：Worker 运行完成后，新的摘要内容会直接覆盖 R2 (书籍) 或 D1 (文章) 中的原有数据。
4.  **完成阶段**：后端返回成功响应，前端接收后执行一次数据拉取动作（Fetch），页面内容静默更新，按钮恢复。

## 3. 详细设计规格

### 3.1 后端接口 (API)
*   **路径**：`/api/reader/summarize`
*   **方法**：`POST`
*   **参数**：
    *   `type`: `'book' | 'article'`
    *   `id`: 资源标识符 (articleId 或 bookId)
    *   `path`: (仅书籍模式) 章节路径
*   **特性**：**同步等待**。接口在逻辑层会等待 Worker 完成 RPC 调用后再返回响应。

### 3.2 存储层处理 (Storage)
*   **不预删模式**：不执行物理删除旧数据或清空字段的操作。
*   **原子覆盖**：Worker 处理完成后，利用存储系统的 `put` (R2) 或 `UPDATE` (D1) 动作实现原子的内容覆盖。

### 3.3 前端逻辑 (Frontend)
*   **Store 动作**：在 `useReaderStore` 中实现 `summarize` 方法，负责 API 调用及后续的刷新工作。
*   **UI 表现**：
    *   在 `Summary` 组件中放置操作入口。
    *   在请求期间，旧摘要保持显示（不显示“摘要生成中”占位符），仅通过按钮状态反馈进度。

## 4. 异常处理
*   **超时风险**：若 AI 响应过慢导致接口超时，前端应提示“生成时间较长，请稍后手动刷新”。
*   **并发控制**：在点击按钮后的 Loading 期间，禁用该按钮，防止重复提交。
