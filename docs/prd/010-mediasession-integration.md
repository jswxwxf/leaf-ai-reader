# PRD-010: MediaSession 系统级媒体控制集成

## 1. 需求背景与目标

目前，阅读器的 TTS 语音朗读功能（基于 `speecher.tsx` 控制器和 `useSpeech.ts` Hook）仅在应用界面可见时受控。为了向用户提供类似 Voice Dream Reader 等顶级播放软件的连续聆听体验，我们需要集成 [MediaSession API](https://developer.mozilla.org/en-US/docs/Web/API/MediaSession_API)。

目标是通过与操作系统底层 API 对接，赋予用户以下能力：
- 使用**键盘原生媒体键**、**蓝牙无线耳机切歌按钮**等硬件事件来控制播放、暂停或跳转。
- 在**移动端锁屏界面**、**通知中心下拉栏**及 **macOS / Windows 的全局媒体控制中心**展示并控制当前朗读进度。
- 解决浏览器音频生命周期管理的痛点，保证系统后台（特别是移动设备息屏锁屏或退至后台时）能持续挂起音频原生控件，提升沉浸式体验。

## 2. 核心功能设计

### 2.1 动作映射 (Action Handlers)

MediaSession 的标准媒体动作将直连至我们现有的事件驱动播放控制逻辑。
- `play` (播放): 触发并代理到业务逻辑的 `play()`。
- `pause` / `stop` (暂停或停止): 触发并代理到业务逻辑的 `stop()`。
- `previoustrack` (上一条曲目内容): 触发并代理到业务逻辑的 `step(-1)`。
- `nexttrack` (下一条曲目内容): 触发并代理到业务逻辑的 `step(1)`。

跳句/跳段的粒度将完全顺承用户当前所选择的朗读模式（Sentence, Paragraph, Article），利用现成 `step` 机制进行游标重置驱动。

### 2.2 元数据展示 (MediaMetadata)

当阅读器处于工作或挂起状态时，系统硬件交互反馈面板需显示直观的基本信息。
- **Title (标题)**: 将显示字段设置为主业务传入的文章全标题。
- 鉴于至简原则，无需额外增加异步开销去轮询提取文章附带头图，亦无需显式赋予复杂的 Artist 属性，默认仅传递唯一的文章标题作为识别锚点即可。

### 2.3 静音保活机制 (Silent Audio Focus / 霸王音频)

若依赖于零散的文本转语音 API，连续音频片段间隙会触发操作系统为主环境回收 MediaSession 控制层。应对策略如下：
- 我们需构建并引入一段无明显时长损耗、且零网络载荷的极短“静音”音频（包含 base64 mp3）。
- 在用户触发任意有效形式的首发 `play()` 时，伴随启动一个不可见的 `<audio>` 标签并通过附加 `loop` 属性实施无限轮换重播。
- 这个隐藏流程将霸占系统的**主音频焦点 (Audio Focus)**。据此，系统底层的锁屏挂件将强制保持存活判定，控制器恒久在线。
- 退火操作将在用户手动释放 `stop()` 事件时同步回收，对保活静音实施销毁或强行 `.pause()`。

## 3. 技术落地方案详述

- **创建隔离层 `useMediaSession` Hook**:
  为了避免将关注维度（BOM 级别的 API 事件注册及音频锁资源调度）紧耦合入负责 TTS 与光标渲染状态的心跳循环，开辟一个独立的 `frontend/src/app/reader/_hooks/use-media-session.ts`。
- **顶层参数界定**:
  - 核心入参需涵盖：`title: string`, `playing: boolean`, 以及回调函数 `play, stop, step`。
- **与业务的结合**:
  由于其实践形态是完全无侵入性质的数据副作用外挂结构，顶层可由对应的 Page 组件或容器内引集并桥接。这样既能规避重渲染缺陷，又能有效守住现有稳定的连续 TTS 阅读机制。
